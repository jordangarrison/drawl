#+title: drawl grammar
#+author: drawl
#+startup: showall

This file is the canonical reference for the drawl source language.
It is the user-facing surface of what =drawl.compiler/compile= accepts.

For pipeline internals and design rationale, see [[file:SPEC.org][SPEC.org]]. For the
reading order =parser → walker → IR → emitter=, see SPEC.org §3–§4. This
document only describes what the user writes.

* Source structure

A =.drawl= source is one or more top-level forms. The compiler implicitly
wraps the whole source in =(do ...)=, so multiple top-level forms are
legal. After macro expansion, exactly *one* =(diagram ...)= form must
remain.

#+begin_src clojure
;; legal top-level shape
(defmacro service [id label tech]            ; user macros (optional)
  (container id label :tech tech))

(diagram "Title"                              ; exactly one
  ,@elements
  ,@relationships)
#+end_src

* Header rule (positional, then keyword/value, then children)

Every form follows the same single-pass header rule:

1. *Positional* args are consumed first, in order. Forms that take
   positional args take the same number every time (e.g. =person= takes
   ~id~ and ~label~).
2. *Keyword/value* pairs are consumed greedily after the positional
   args, until the first form that is not a keyword.
3. *Children* are everything that remains. Children may only nest where
   the spec allows it (system → container → component).

This is a single forward pass, no backtracking. The walker enforces it
in =drawl.walker/split-attrs=.

#+begin_src clojure
;; container takes 2 positional, then attrs, then children
(container web "Web App"            ; positional: id label
  :tech "Phoenix"                   ; attr
  :role :default                    ; attr
  (component auth "Auth"))          ; child
#+end_src

* Top-level form

** =(diagram TITLE form*)=

The root. ~TITLE~ is a string. Body is any mix of element forms and
relationship forms.

#+begin_src clojure
(diagram "Internet Banking"
  (person customer "Customer")
  (system bank "Banking System"
    (container web "Web App")
    (container api "API")
    (container db "Database" :role :database))
  (-> customer web "uses")
  (-> web api))
#+end_src

* Element forms

| Form                                         | Where it nests                  | Notes                              |
|----------------------------------------------+---------------------------------+------------------------------------|
| =(person id label)=                          | inside =diagram=                | actor / role / external person     |
| =(system id label children*)=                | inside =diagram= or =system=    | a system; nested = system-of-systems |
| =(container id label children*)=             | inside =system=                 | a container; can host components   |
| =(component id label)=                       | inside =container=              | leaf                               |

All four accept the [[#attributes][attributes]] below.

** ID rules

- IDs are bare symbols.
- They live in *one flat global namespace*. Duplicates are a hard error.
- Qualified =system/component= syntax is reserved for future use; today
  every ID must be unique across the whole diagram.
- The same ID is used as the relationship endpoint (see =->= below).

* Attributes
:PROPERTIES:
:CUSTOM_ID: attributes
:END:

Keyword/value pairs after the positional args, before any children.

| Attr          | Type      | Effect                                                   |
|---------------+-----------+----------------------------------------------------------|
| =:tech=       | string    | technology tag, rendered in the label                    |
| =:role=       | keyword   | shape hint (e.g. =:database= → cylinder in dot)          |
| =:external=   | boolean   | flips suffix to =_Ext= in mermaid C4                     |
| =:style=      | string    | passed through to the backend (dot =style==, etc.)       |

Unknown attrs are accepted by the walker and silently ignored by the
emitter. This is intentional — the surface stays open for new attrs to
be introduced without breaking older diagrams.

* Relationships

** Two-symbol explicit form

| Form                                | Direction       |
|-------------------------------------+-----------------|
| =(-> a b)=                          | a → b           |
| =(-> a b "label")=                  | a → b, labelled |
| =(-> a b "label" :tech "HTTPS")=    | a → b, with tech |
| =(<-> a b "label")=                 | a ↔ b           |

Endpoints (~a~, ~b~) must resolve to declared element IDs. Unresolved
endpoints are a hard error.

** Implicit-=from= inside element bodies

When =(-> ...)= or =(<-> ...)= appears inside an element's body
(=person= / =system= / =container= / =component=), a single-symbol
arity means "from this element":

#+begin_src clojure
(container api "API"
  (-> db "reads")               ; from = api, implicit
  (-> stripe "charges" :tech "REST")
  (-> spa api "delivers"))      ; two symbols = explicit, still legal
#+end_src

Outside an element body, the single-symbol form is a hard error.

Self-loops are allowed: =(-> a a)= and =(container a (-> a))= both
produce =a → a=.

** =(=> a b c …)= --- chain with vector fan

Each adjacent pair in the chain produces an edge. A vector at any
position is a parallel set; adjacent vectors yield the full
cross-product. A trailing string + keyword/value pairs apply to every
edge in the chain.

#+begin_src clojure
(=> a b c d)                              ; a→b, b→c, c→d
(=> a [b c] d)                            ; a→b, a→c, b→d, c→d
(=> [a b] [c d])                          ; 4 edges, cross-product
(=> customer api "via HTTPS" :tech "HTTPS")
#+end_src

Chain must have at least two nodes; fan vectors must be non-empty and
contain only symbols. Per-edge labels are not expressible --- use
=(-> ...)= for those.

The trailing label/attrs apply uniformly to every edge in the chain
(including cross-product fan-out). If you need different labels per
hop, write each as =(-> ...)= directly; mixing per-hop labels into
=>= would reintroduce position-dependent parsing in a form that
already does fan + cross-product.

** =(with-attrs {…} body…)= --- defaults supplier

Wraps any body. Defaults flow into every edge constructed in the
subtree, no matter how deeply nested (across element boundaries, into
=->=, =<->=, =>=, or macro expansions). Edge attrs win on conflict;
inner =with-attrs= overrides outer.

#+begin_src clojure
(with-attrs {:tech "gRPC"}
  (-> api auth)
  (-> api billing)
  (=> customer [web mobile] api))
#+end_src

First argument must be a map literal; otherwise the form is rejected
at walk time. Empty body is allowed (no edges produced).

=with-attrs= reaches edges declared inside elements nested in its body
--- this is intentional. Lexical scope is the contract; if you want
locality, wrap a smaller subtree.

#+begin_src clojure
(with-attrs {:tech "gRPC"}
  (container api "API"
    (-> auth "verify")        ; gets :tech "gRPC"
    (-> catalog "lookup")))   ; gets :tech "gRPC"
#+end_src

* Built-in macros

The walker seeds a per-compile macro registry with these shorthands.
They expand into the underlying primitives.

| Macro                       | Expands to                                    |
|-----------------------------+-----------------------------------------------|
| =(webapp id label)=         | =(container id label :tech "Web")=            |
| =(rest-api id label)=       | =(container id label :tech "REST API")=       |
| =(postgres-db id label)=    | =(container id label :role :database :tech "PostgreSQL")= |
| =(redis-cache id label)=    | =(container id label :role :database :tech "Redis")=      |

Built-ins live in =drawl.macros/builtins= and are templates, not
=eval=-time forms — see the user-macro section.

* User macros

#+begin_src clojure
(defmacro NAME [params...] body)
#+end_src

User =defmacro= forms appear at the *top level* of a source, outside the
=diagram= form. They register into the per-compile =:macros= context and
override built-ins by the same name (with a warning).

The expansion is *plain symbol substitution*: the body is walked, and
any leaf symbol that matches a parameter is replaced with the
caller-supplied form. There is no =eval=, no syntax-quote, no =~= or
=~@=. =&= captures the rest as a splice site.

#+begin_src clojure
(defmacro service [id label tech]
  (container id label :tech tech))

(defmacro tier [id label & children]
  (system id label & children))

(diagram "Hello"
  (service api "API" "Phoenix")
  (tier infra "Infra"
    (postgres-db db "Main DB")))
#+end_src

* Levels

Levels are *inferred*, not declared. The deepest =:kind= present in the
walked IR wins:

#+begin_example
component  ; deepest
container
context    ; shallowest
#+end_example

A single source can be filtered to a lower level via
=(drawl.ir/at-level :context ir)= in the JVM API. The browser SPA
currently always renders at the inferred level.

* Backend mapping (for reference)

See SPEC.org §6 for the full mapping. Quick highlights:

- *graphviz dot* — =:role :database= → =shape=cylinder=. Boundaries
  become =subgraph cluster_N=. =:style= passes through verbatim.
- *mermaid C4* — header per level (=C4Context= / =C4Container= /
  =C4Component=). =:external= flips =Foo= → =Foo_Ext=. Systems and
  containers with children become =System_Boundary= /
  =Container_Boundary=. =:role :database= → =*Db= variant;
  =:role :queue= → =*Queue=. Edges where an endpoint is a boundary
  are rewritten to the boundary's first leaf descendant in
  declaration order (mermaid C4 refuses boundary endpoints
  directly). Edge =:style= and =:role= on Person/System are silently
  ignored — no native equivalent.

* Errors

All compiler errors are =ex-info= with =ex-data= keys ~:type~ (one of
=:parse-error=, =:walk-error=, =:emit-error=), ~:position~ (when
available), ~:message~. The browser SPA shows ~:message~ in the red
bar; the JVM API surfaces the full =ex-info=.
