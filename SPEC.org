#+TITLE: drawl
/A Lisp for diagrams. C4-aligned, browser-native, CLI-friendly./

--------------

* 0. About the name
=drawl= reads two ways. As a casual user reads it, it's the slow, considered cadence you'd give to careful work --- diagrams take their time. For Lispers, it's DRAW with the L of /Lisp/ sliding in on the shared letter, with a faint nod to Lisp's etymological origin as a "lithp" --- the speech-impediment joke from the language's earliest days.

Optional backronym, for when the project ever needs one: *D*iagrams *R*endered *A*s *W*alked *L*ists --- a literal description of what the compiler does (parse forms, walk the IR, emit diagrams).

File extension is =.drawl=. CLI command is =drawl=. Tagline: /"Diagrams, with a lisp."/

--------------

* 1. Goals & Non-Goals
** Goals
- A minimal Lisp surface for declaring diagrams. Forms nest, edges reference elements by name, attributes are keyword-value pairs.
- C4 model alignment as a first-class concern: =person=, =system=, =container=, =component= are built-in element types.
- Multi-target compilation from a single codebase: browser SPA, Babashka CLI, JVM library.
- Pluggable backends. Render to mermaid C4, graphviz dot, or Excalidraw today; PlantUML or Structurizr DSL later if useful.
- Live editing in the browser, sub-150ms from keystroke to rendered SVG.

** Non-Goals
- Replacing Structurizr for serious enterprise C4 authoring. drawl is a faster, more hackable alternative for personal and small-team use.
- General-purpose graph layout. Layout is delegated to backends (graphviz, mermaid).
- Code-level (C4 L4) diagrams. Out of scope for v1.
- Rich text formatting in labels. Plain strings only.

--------------

* 2. Language Reference
** 2.1 Lexical Structure
Source is parsed by Clojure's reader --- =clojure.edn/read-string= on JVM/Babashka, =cljs.reader/read-string= in the browser. The full source is wrapped in an implicit =(do ...)= form so multiple top-level forms (e.g. macro definitions plus a diagram) are legal.

| Token kind | Examples                        | Notes                            |
|------------+---------------------------------+----------------------------------|
| Symbol     | =web=, =customer=, =order-service=    | Element names, edge endpoints    |
| String     | ="Web Application"=               | Titles, descriptions, label text |
| Keyword    | =:tech=, =:database=, =:external=     | Attribute keys, enum values      |
| Number     | =42=, =0.5=                         | Used in some attribute values    |
| List       | =(component web :tech "Phoenix")= | Forms                            |
| Comment    | =; comment to EOL=                | Standard Lisp line comments      |

** 2.2 Form Grammar
#+begin_example
form        := (head head-args... header... children...)
head        := symbol
head-args   := positional arguments specific to the head
                (e.g. element ID symbol, edge endpoints)
header      := strings (≤2: title, description) and :keyword value
                pairs, in any order
children    := nested list forms
#+end_example

*Header parsing rule*: after the head's required positional arguments, the walker peels strings (capped at 2 — bound to title and description in the order encountered) and keyword/value pairs (last-write wins on duplicate keys) in any interleaved order. The first non-string, non-keyword form ends the header and starts children. Single pass, no backtracking.

This relaxed rule makes template macros (§2.6) trivially composable: a macro that injects =:tech "Web"= before splicing user opts can be called with a positional title, and the walker still binds the title correctly.

** 2.3 Forms
*** =diagram=
#+begin_src lisp
(diagram TITLE? :attrs... ELEMENTS...)
#+end_src

The top-level form. Required. Exactly one per source after macro expansion.

- =TITLE= (optional string): displayed as the diagram title.
- Common attributes: =:notation= (=:c4= default, =:plain=).
- =ELEMENTS=: any number of =person=, =system=, edges at the top level, or macro-expanded forms.

*** =person=
#+begin_src lisp
(person ID TITLE? DESCRIPTION? :attrs...)
#+end_src

A C4 actor --- renders as a stick figure or person icon depending on backend.

- =ID= (symbol, required): used to reference this person in edges.
- =TITLE= (optional string): display name.
- =DESCRIPTION= (optional string): role description, second positional string.
- Attributes: =:external= (boolean).

*** =system=
#+begin_src lisp
(system ID TITLE? DESCRIPTION? :attrs... CHILDREN...)
#+end_src

A C4 software system. Renders as a leaf in context views, or as a boundary containing containers in container views.

- =ID= (symbol, required).
- =TITLE=, =DESCRIPTION= (optional strings).
- Attributes: =:external= (boolean).
- =CHILDREN=: zero or more =container= forms or edges scoped to this system.

*** =container=
#+begin_src lisp
(container ID TITLE? DESCRIPTION? :attrs... CHILDREN...)
#+end_src

A C4 container. Renders as a leaf in container views, or as a boundary in component views.

- =ID= (symbol, required).
- =TITLE=, =DESCRIPTION= (optional strings).
- Attributes: =:tech= (string), =:role= (=:database=, =:queue=, =:browser=, =:lambda=, =:cache=), =:external= (boolean).
- =CHILDREN=: zero or more =component= forms or edges scoped to this container.

*** =component=
#+begin_src lisp
(component ID TITLE? DESCRIPTION? :attrs...)
#+end_src

A C4 component. Always a leaf.

- =ID= (symbol, required).
- =TITLE=, =DESCRIPTION= (optional strings).
- Attributes: =:tech=, =:role=, =:external=.

*** =->= (edge)
#+begin_src lisp
(-> FROM TO DESCRIPTION? :attrs...)
#+end_src

A directed relationship.

- =FROM=, =TO= (symbols, required): names of any defined element.
- =DESCRIPTION= (optional string): label on the edge.
- Attributes: =:tech= (string, often shown as a smaller label), =:style= (=:solid= default, =:dashed=, =:dotted=), =:label= (alternate to positional description).

*** =<->= (bidirectional edge)
#+begin_src lisp
(<-> A B DESCRIPTION? :attrs...)
#+end_src

Convenience for two-way relationships. Renders as the backend's bidirectional edge primitive (=Rel_Bi= in mermaid C4, =dir=both= in dot).

*** Edge-surface extensions

Three additive mechanisms layer onto =(->)= / =(<->)= without changing the IR. See [[file:docs/specs/2026-05-13-edge-syntax-design.md][docs/specs/2026-05-13-edge-syntax-design.md]] for the full design discussion; the user-facing reference lives in [[file:GRAMMAR.org][GRAMMAR.org]] §Relationships.

- *Implicit-=from= on =(->)= and =(<->)= inside element bodies.* When an edge form appears as a child of =person= / =system= / =container= / =component=, a single-symbol arity (=(-> b)=) means "from the enclosing element to =b=". The two-symbol arity (=(-> a b)=) is unchanged. Outside an element body, single-symbol is a walk error.
- *=(=> a b c …)= --- chain with vector fan.* New head. Each adjacent pair of chain positions produces an edge; a vector at any position contributes a parallel set, and adjacent vectors take the full cross-product. Trailing string + kw/val pairs apply to every edge.
- *=(with-attrs {…} body…)= --- defaults supplier.* Wraps any body. The supplied map merges UNDER each edge's own attrs (edge attrs win) for every edge produced anywhere in the subtree, including across element boundaries and macro expansions. Nestable; inner overrides outer.

These desugar entirely in the walker; emitters and =drawl.ir/validate= are untouched.

Cluster-edge rendering (=compound=true= + =ltail=/=lhead=) is a
graphviz/dot concern. Mermaid C4 has no equivalent of
=ltail=/=lhead= /and/ rejects edges where an endpoint is itself a
=System_Boundary= or =Container_Boundary= (the layout pass throws
=Cannot read properties of undefined (reading 'x')=). The mermaid
emitter mirrors dot's "first leaf in declaration order" anchor
contract: when an edge endpoint is a boundary, the emitter rewrites
it to the first leaf descendant. The boundary itself is not
highlighted visually — the edge just attaches to the chosen interior
leaf — but the diagram renders.

** 2.4 Attribute Reference
| Attribute    | Applies to                 | Type    | Notes                                        |
|--------------+----------------------------+---------+----------------------------------------------|
| =:tech=        | container, component, edge | string  | Technology label                             |
| =:role=        | container, component       | keyword | =:database=, =:queue=, =:cache=, =:browser=, =:lambda= |
| =:external=    | person, system, container  | boolean | Renders as out-of-scope                      |
| =:description= | any element                | string  | Alternative to positional description        |
| =:style=       | edge                       | keyword | =:solid= (default), =:dashed=, =:dotted=           |
| =:label=       | edge                       | string  | Alternative to positional description        |
| =:notation=    | diagram                    | keyword | =:c4= (default), =:plain=                        |

Unknown attributes are accepted by the parser and walker, and ignored by the emitter (with optional warning). This lets user macros introduce custom attributes without changing the core.

** 2.5 References & Name Resolution
- Bare symbols in =from=/=to= positions of edges are references.
- References resolve via a flat global namespace. Every =person=/=system=/=container=/=component= defines a name in one table.
- Defining the same name twice is a hard error.
- Referring to an undefined name is a hard error.
- Containers and systems ARE referenceable. An edge between two systems renders as a boundary-to-boundary edge --- handled per backend (mermaid C4 supports natively, graphviz uses cluster =lhead=/=ltail=).
- No lexical scoping. Qualified references (e.g. =system/component=) are reserved syntax for future use, not implemented in v1.

** 2.6 Macros
Shorthand expands to canonical forms during walking. The same registry-driven
mechanism powers both built-in shorthand and user-defined macros — there is
exactly one macro system.

*** 2.6.1 Built-ins
v1 ships four pre-seeded templates. Each may be overridden by a user
=(defmacro ...)= of the same name (with a warning).

| Macro                      | Expands to                                                |
|----------------------------+-----------------------------------------------------------|
| =(webapp NAME OPTS...)=      | =(container NAME :tech "Web" OPTS...)=                      |
| =(postgres-db NAME OPTS...)= | =(container NAME :role :database :tech "Postgres" OPTS...)= |
| =(redis-cache NAME OPTS...)= | =(container NAME :role :cache :tech "Redis" OPTS...)=       |
| =(rest-api NAME OPTS...)=    | =(container NAME :tech "REST API" OPTS...)=                 |

*** 2.6.2 User-defined Macros
Top-level =(defmacro NAME [params...] body)= forms register a template into
the per-compile registry. Macros must appear *outside* the diagram (single
pass, no recursion in registration). Convention: place all =defmacro= forms
before the =(diagram ...)= form.

#+begin_src lisp
(defmacro mongo-db [name & opts]
  (container name :role :database :tech "MongoDB" opts))

(diagram (system shop (mongo-db users "Users")))
#+end_src

*Substitution.* Inside the body, fixed params bind by symbol — anywhere the
param symbol appears as a leaf, it's replaced with the bound value. The
symbol after =&= captures the rest of the call-site args; when it appears as
a leaf inside any list in the body, that list is spliced in. There is no
=eval=, no =~=/=~@= reader markers, no compile-time computation: just
literal forms with parameter substitution.

*Override.* If a =defmacro= name matches a built-in or a previously-defined
user macro, the new template wins (last definition); a warning is printed
to standard error.

*Limits in v1.* No nested =defmacro= inside another macro body; the registry
is built before any walking begins. Macro bodies cannot call themselves
(infinite recursion is the user's responsibility). Built-in shorthand may
also be added directly to =drawl.macros/builtins= for stack-neutral
extensions.

* 3. Semantic Model
** 3.1 Intermediate Representation
The walker produces a single map:

#+begin_src clojure
{:title         string?
 :level         #{:context :container :component}
 :attrs         map?
 :elements      [Element]
 :relationships [Edge]}

Element :=
  {:kind        #{:person :system :container :component}
   :id          symbol?
   :title       string?
   :description string?
   :attrs       map?
   :children    [Element]    ; only for :system, :container
   :edges       [Edge]}      ; edges scoped to this element

Edge :=
  {:kind            :edge
   :from            symbol?
   :to              symbol?
   :bidirectional?  boolean?
   :description     string?
   :attrs           map?}
#+end_src

Every element has the same envelope; varying only in =:kind=. Containment lives in =:children=, which lets one recursive walker handle systems-with-containers and containers-with-components uniformly.

** 3.2 Level Inference
Level is computed in one tree walk by taking the deepest =:kind= encountered:

- Any =:component= element → =:level :component=
- Else any =:container= element → =:level :container=
- Else → =:level :context=

The level controls backend rendering: which mermaid C4 header to emit (=C4Context= / =C4Container= / =C4Component=), which boundary types are valid.

The same source can be filtered to lower-level views via =(drawl.ir/at-level :context ir)=, which strips containers and components and leaves only systems and people. This means one source can serve multiple zoom levels.

** 3.3 Validation
The walker enforces:

- Exactly one top-level =(diagram ...)= form (after macro expansion).
- All form heads are valid: =person=, =system=, =container=, =component=, =->=, =<->=, or a registered macro.
- Element nesting is sound: components only inside containers, containers only inside systems, systems only inside diagrams or inside other systems (system-of-systems / landscape views).
- All edge endpoints resolve.
- All required positional arguments are present.

Errors are returned as =ex-info= exceptions with =ex-data= carrying =:type= (=:parse-error= / =:walk-error= / =:emit-error=), =:position= (line/col when available), and =:message=.

** 4.3 Walker Implementation Sketch
Multimethod dispatch on head symbol --- keeps the walker open for extension:

#+begin_src clojure
(defmulti walk-form (fn [form _ctx] (first form)))

(defmethod walk-form 'diagram [form ctx]
  (let [[_ & rest]                 form
        [[title _] attrs children] (split-header rest)]
    {:title         title
     :attrs         attrs
     :elements      (mapv #(walk-form % ctx) children)}))

(defmethod walk-form 'system    [form ctx] (walk-element :system form ctx))
(defmethod walk-form 'container [form ctx] (walk-element :container form ctx))
(defmethod walk-form 'component [form ctx] (walk-element :component form ctx))
(defmethod walk-form 'person    [form ctx] (walk-element :person form ctx))

(defmethod walk-form '->  [form ctx] (walk-edge false form))
(defmethod walk-form '<-> [form ctx] (walk-edge true  form))
#+end_src

Macros (built-in *and* user-defined) live in =(:macros ctx)= as a registry of
=symbol → {:params {:fixed :rest} :body form}=. The =:default= defmethod
consults this registry: if the head matches, parameters are bound to call-site
args, the body is substituted, and =walk-form= recurses on the expansion. If
the head is not in the registry, the walker throws.

#+begin_src clojure
(defmethod walk-form :default [form ctx]
  (if-let [tpl (get-in ctx [:macros (first form)])]
    (walk-form (expand tpl form) ctx)
    (throw (ex-info "Unknown form head" {:type :walk-error :form form}))))
#+end_src

=drawl.compiler/parse= seeds the registry with =drawl.macros/builtins=, then
folds in any top-level =(defmacro ...)= forms (last-write wins, with a
warning).

*Walker contract additions (edge-surface).* Two ctx keys carry
edge-surface state through the walk; one structural rule lets a single
form produce many IR nodes. Full rationale in
[[file:docs/specs/2026-05-13-edge-syntax-design.md][docs/specs/2026-05-13-edge-syntax-design.md]].

- =:current-element= — symbol id of the nearest enclosing element. =walk-element= assocs this into ctx before descending into children; =walk-edge= reads it for the implicit-=from= path.
- =:wrap-attrs= — merged map of attribute defaults from enclosing =with-attrs= forms. =walk-with-attrs= updates it (inner overrides outer); every edge-constructing path (=walk-edge=, =walk-chain=) merges it UNDER the edge's own attrs so edge attrs win on conflict.
- =walk-children= flattens vector returns. =walk-form= may now return either a single IR node or a vector of nodes (=(=> ...)= yields N edges; =(with-attrs ...)= yields M children). =walk-children= mapcat-splices vector returns into its result list before partitioning into =:elements= and =:relationships=.

** 4.4 Reader Compatibility
=clojure.edn/read-string= (JVM) and =cljs.reader/read-string= (browser) are nearly but not bug-for-bug identical. Specifically watch for tagged literals and namespaced keywords. The portability test suite (§7.3) loads the same fixture file through both runtimes and asserts identical IR output.

** 6.3 Excalidraw
For hand-drawn-feel diagrams that round-trip through Excalidraw or excalidraw.com --- the user can edit, restyle, or annotate after generation. Output is an Excalidraw scene JSON document; conventional file extension is =.excalidraw=.

*Layout strategy.* Excalidraw has no auto-layout --- every element needs absolute =x=/=y=/=width=/=height=. The emitter computes layout directly from the IR using a recursive containment pass: leaves get a fixed box sized to their label; a parent (system/container with children) lays its children out in a row (wrapping to a grid above a threshold), pads, and is sized to enclose them; siblings at the diagram level flow horizontally. Edges are straight segments between endpoint shape centers, with Excalidraw bindings handling the visual clearance. No external layout tool is required and the same code runs identically on JVM, Babashka, and the browser.

This stays within the §1 non-goal of "general-purpose graph layout" --- the IR is already a containment tree, and rendering a containment tree to a hand-drawn surface is bbox arithmetic, not graph layout. If edge crossings become ugly on a particular diagram, that is what Excalidraw's editor is for; round-trip is the design goal.

| Element                       | Excalidraw output                                                                              |
|-------------------------------+------------------------------------------------------------------------------------------------|
| =:person=                     | =ellipse= with text label, sketchy =roughness: 1=                                                  |
| =:system= (leaf)                | =rectangle= with rounded corners, text label                                                     |
| =:system= with children         | =frame= holding child elements (children carry =frameId=); =name= = system title                       |
| =:container=                    | =rectangle= with title + =:tech= text bound via =boundElements=                                      |
| =:role :database=               | =ellipse= (cylinder approximation; Excalidraw has no native cylinder)                            |
| =:role :queue=                  | =rectangle= with dashed =strokeStyle=                                                              |
| =:component=                    | =rectangle=                                                                                      |
| =:external=                     | =strokeStyle: dashed= on the surrounding shape                                                   |
| =(-> a b "desc" :tech "X")=     | =arrow= with =startBinding=/=endBinding= to element ids; label as a bound =text= element at midpoint |
| =(<-> a b ...)=                 | =arrow= with both =startArrowhead= and =endArrowhead= set                                            |
| Edge =:style :dashed= / =:dotted= | maps to =strokeStyle: dashed= / =dotted=                                                           |

The emitted JSON conforms to Excalidraw's =type: "excalidraw"= scene format (top-level ={type, version, source, elements, appState, files}=). Element ids are stable hashes of the drawl source ID so re-compilation preserves identity for users who have manually tweaked a scene and re-import.

Boundaries (=:system= with children, and future =:container= boundaries) use Excalidraw =frame= elements rather than =groupIds=. A frame carries a visible labeled outline and owns its children via ~frameId~ on each child --- which matches the IR's boundary semantics. =groupIds= would encode only "these elements happen to be selected together," losing the named-boundary meaning that already lives in the IR.

The emitter matches the existing backend signature: ~(drawl.emit.excalidraw/emit ir) → string~. Pure IR-in, JSON-string out, no IO, no external process. The dispatch in =drawl.compiler/emit= is the same =case= shape as the other backends. Internally the emitter is two passes: a layout pass walks the IR and returns a flat map ~{id → {:x :y :w :h}}~; an emit pass walks the IR again with that layout in hand and builds the Excalidraw element list. Both passes are pure data transforms.

This means =:dot= and =:excalidraw= are independent backends with no shared layout step. =:dot= emits dot source and lets graphviz (or viz.js, downstream of the SPA) lay out and render; =:excalidraw= emits Excalidraw scene JSON with layout baked in and lets the Excalidraw app render. Same IR, two emitters, no internal coupling between them.

A worked example of this transform --- =03-bank-containers.drawl= compiled to dot, fed through =dot -Tjson=, and hand-translated into an =.excalidraw= scene --- lives at =test/fixtures/excalidraw/= with accompanying =TRANSFORM.org= notes (units, Y-flip, binding rules, gotchas). Future emitter snapshot tests diff against that fixture.

* 7. Testing
** 7.1 Walker Tests
For each form, assert IR shape and presence of expected fields. Cover edge cases: missing required fields, duplicate names, unresolved refs, level inference at all three levels, attribute parsing boundary (the moment children start).

** 7.2 Emitter Tests
Snapshot tests. Given a fixture IR, assert the emitted string matches expected output. Maintain a small library of canonical examples --- the C4 "Internet Banking" model is a good baseline because it exercises every element kind and both internal and external systems.

** 7.3 Portability Tests
The same =.cljc= fixture should produce identical IR when run through Clojure (JVM) and ClojureScript (via shadow-cljs =:node-test= build). Catches reader divergences early. Wire into CI so it runs on every push.

** 7.4 End-to-End
- Browser SPA: Playwright tests confirming the editor + render pipeline. Type a known good diagram, assert SVG appears in the render pane.
- CLI: shell-based --- =bb drawl compile fixture.drawl | diff - expected.txt=.

** v0.6 --- Editor support
drawl is its own Lisp, not Clojure --- the shared reader is an implementation detail, not a language identity. Editor mode therefore derives from =lisp-mode=, *not* =clojure-mode=, so users don't inherit Clojure-specific behavior (cider hooks, =deps.edn= project detection, =defn=-style indentation, namespace assumptions) on =.drawl= buffers.

- =editor/emacs/drawl-mode.el= --- =define-derived-mode drawl-mode lisp-mode=. Registers =.drawl= in =auto-mode-alist=. Ships:
  - A small syntax table extending =lisp-mode-syntax-table= for Clojure-reader tokens drawl actually uses: =:keyword=, =[vector]=, ={map}=, =;= line comments. (No tagged literals, no metadata reader macros --- drawl source doesn't use them.)
  - Font-lock for the built-in heads (=diagram=, =person=, =system=, =container=, =component=, =->=, =<->=) and the shorthand macros (=webapp= et al.).
  - Plain Lisp indentation (every form indented as a function call). drawl has no =let=, no =defn=, so the simple rule is correct.
  - =drawl-mode-hook= recommends =paredit-mode= or =smartparens-strict-mode= for structural editing (slurp/barf), since we don't inherit clojure-mode's bindings.
  - =M-x drawl-compile= --- shells to =drawl compile= on the current buffer, opens output in a side buffer.
  - =M-x drawl-render= --- shells to =drawl render= on the current buffer, opens the SVG inline.
  - =M-x drawl-lint= --- runs =drawl lint=, surfaces errors via flymake/flycheck.
- =editor/vscode/= --- TextMate grammar + extension that wraps the same CLI calls. Lower priority.
- Optional =drawl lsp= subcommand: emits diagnostics, completion (element ids in edge =from=/=to= positions, attribute keys, valid =:role= values), and go-to-definition for element ids. Built on lsp4clj. Both Emacs (=eglot=/=lsp-mode=) and VS Code wire to it. With LSP carrying the semantic load, the static =drawl-mode= can stay tiny.
- Tree-sitter grammar lands here only if the LSP route doesn't materialise --- LSP diagnostics + lisp-mode highlighting cover most of what tree-sitter would buy.

* 9. Open Questions
A few decisions left to make before v0.1 starts:

- *UIx vs Reagent.* Either works. UIx is the more modern feel; Reagent has more documentation. No wrong answer.
- *Examples library.* Ship a small set (Internet Banking, three-tier web app, microservices, Phoenix LiveView app) so the editor opens to something interesting rather than a blank page.
- *Error rendering.* When a parse error happens mid-keystroke, do you keep showing the last good render, or clear it? Recommend the former --- less flicker.
